# "a2a-agent-card-skill-governor" Policy

This is the policy-implementation half of the **A2A Agent Card Skill Governor** policy. It compiles to a `wasm32-wasip1` module loaded by Omni Gateway at runtime.

For the full policy behavior, configuration reference, and worked examples, see the [root README](../README.md).

The policy was created with the Omni Gateway Policy Development Kit (PDK). For the complete PDK documentation, see [PDK Overview](https://docs.mulesoft.com/pdk/latest/policies-pdk-overview).

## Implementation notes

This is a **response-shaping** policy. The request filter does minimal work — it classifies the request into a card surface + A2A variant (`detect.rs`), reads caller identity from the `Authentication` injectable for the extended surface, and threads a `CardContext` forward via `RequestData`. It never blocks. The response filter (`lib.rs`) locates the `AgentCard` (bare body, or JSON-RPC `result`), runs the rule engine (`governor.rs`) over `skills[]`, and writes the reshaped card back.

Two structural points worth knowing when working on the code:

- **Outbound injection.** `gcl.yaml` declares `injectionPoint: outbound`. In integration tests the policy therefore binds via `ApiConfig`'s `.outbound_policies(...)`, **not** `.policies(...)` — an outbound policy cannot bind to the inbound slot of an `ApiInstance` (the gateway rejects it). This is the key deviation from an inbound policy template.
- **Split headers→body response flow, body-only fail-closed.** The response filter drops the stale `content-length` in the headers phase, then reads and rewrites the body in the body phase. The combined headers+body state is deliberately not used on the response leg — it hangs (Envoy 504) on the targeted runtime. A consequence: on fail-closed, the `:status` is already committed, so the policy replaces only the body with a surface-appropriate error envelope and leaves the status as the upstream sent it. See `fail_closed_body` / `response_filter` doc comments.

## Make command reference
This project has a Makefile that includes different goals that assist the developer during the policy development lifecycle.

*For more information about the Makefile, see [Makefile](https://docs.mulesoft.com/pdk/latest/policies-pdk-create-project#makefile).*

### Setup
The `make setup` goal installs the Policy Development Kit internal dependencies for the rest of the Makefile goals.

*For more information about `make setup`, see [Setup the PDK Build environment](https://docs.mulesoft.com/pdk/latest/policies-pdk-create-project#setup-the-pdk-build-environment).*

### Build asset files
The `make build-asset-files` goal fetches the latest policy definition from Exchange and regenerates `src/generated/config.rs` from the GCL schema.

*For more information about creating a policy definition, see [Defining a Policy Schema Definition](https://docs.mulesoft.com/pdk/latest/policies-pdk-create-schema-definition).*

### Build
The `make build` goal compiles the WebAssembly binary of the policy.
Since the source code must be in sync with the policy definition configuration, make sure you've at least run `make build-asset-files` once before compiling.

*For more information about `make build`, see [Compiling Custom Policies](https://docs.mulesoft.com/pdk/latest/policies-pdk-compile-policies).*

### Run
The `make run` goal runs the current build of the policy in a Docker-containerized local Omni Gateway. The `playground/config` directory must contain the files required to run the policy in Local Mode:

- A `registration.yaml` file generated by performing an Omni Gateway registration in Local Mode. If you already have an instance registered in Local mode, you can reuse that registration file and copy it into `playground/config`.
  Otherwise, to complete the registration we recommend using the Anypoint Platform:
    1. Go to `Runtime Manager`.
    2. Navigate to the `Omni Gateway` tab.
    3. Click the `Add Gateway` button.
    4. Select `Docker` as your OS and copy the registration command, replacing `--connected=true` with `--connected=false`.
    5. Paste the command and run it in the `playground/config` directory.
- An `api.yaml` file with the desired policy configuration. The repository ships with an example combining a deny rule, a glob deny, and a rewrite.

For a step-by-step walkthrough with `curl` examples against all three card surfaces, see the local-testing skill at `.claude/test-a2a-agent-card-skill-governor-locally/SKILL.md`.

*For more information about `make run`, see [Debugging Custom Policies Locally with PDK](https://docs.mulesoft.com/pdk/latest/policies-pdk-debug-local).*

### Test
The `make test` goal runs unit tests and integration tests. Integration tests are placed in the `tests` directory.

Integration tests require a `tests/config/registration.yaml` produced via `anypoint-cli-v4 registration create-local` (it is gitignored). They spin up a containerized Omni Gateway plus an `httpmock` upstream that serves a fixed AgentCard on each card surface, and assert on the governed response.

Convenience targets:
- `make test-unit` — run the unit tests only (no Docker required).
- `make test-one TEST=<name>` — run a single test by name.

*For more information about writing integration tests, see [Writing Integration Tests](https://docs.mulesoft.com/pdk/latest/policies-pdk-integration-tests).*

### Publish
The `make publish` goal publishes the policy asset in Anypoint Exchange, in your configured Organization.

Since the publish goal is intended to publish a policy asset in development, the _assetId_ and name published will explicitly say `dev`, and the versions published will include a timestamp at the end of the version. Eg.
- groupId: your configured organization id
- visible name: _{Your policy name} Dev_
- assetId: _{your-policy-asset-id}-dev_
- version: _{your-policy-version}-20230618115723_

*For more information about publishing policies, see [Uploading Custom Policies to Exchange](https://docs.mulesoft.com/pdk/latest/policies-pdk-publish-policies).*

### Release
The `make release` goal also publishes the policy to Anypoint Exchange, but as a ready for production asset. In this case, the groupId, visible name, assetId and version will be the ones defined in the project.

*For more information about releasing policies, see [Uploading Custom Policies to Exchange](https://docs.mulesoft.com/pdk/latest/policies-pdk-publish-policies).*

### Policy Examples

The PDK provides a set of example policy projects to get started creating policies and using the PDK features. To learn more about these examples see [Custom policy Examples](https://docs.mulesoft.com/pdk/latest/policies-pdk-policy-templates).
